WS6 migration runbook

หลักการ
- ไม่แตะ config.py, db.py, web/cycle service เดิมของ phase 1
- ไม่เขียนกลับ production DB
- patch แบบ additive เท่านั้น
- ไม่มีการใช้ root ใน runtime script บน production
- backup ใช้ mon_backup@127.0.0.1 เท่านั้น
- restore database import เป็น DBA-controlled procedure แยกจาก runtime users

Pre-check
1) ยืนยัน Phase 1 + WS1-WS5 ผ่าน UAT และ route เดิมยังทำงาน
2) ยืนยัน monitor_servers กลับไปใช้ master/slave จริงแล้ว
3) เก็บ baseline source code ที่รันอยู่จริง
4) รัน scripts/preflight_check.sh
5) รัน scripts/backup_monitor.sh แล้วจด path backup
6) รัน scripts/config_diff.sh เพื่อเก็บ drift ปัจจุบัน

Deploy order
1) unzip patch ws6 ทับลง /opt/mysql-repl-monitor
2) chmod +x scripts/*.sh
3) systemctl restart mysql-repl-monitor-web.service
4) รัน scripts/postflight_check.sh
5) รัน scripts/drill_restore_test.sh <latest_backup_dir>
6) ให้ DBA ทำ restore drill แยกตาม docs/WS6_BACKUP_RESTORE_DRILL.txt
7) ถ้าผ่าน ค่อย enable backup timer

Rollback
1) หยุดเฉพาะ service/timer ที่เพิ่มจาก ws6
2) คืนไฟล์ docs/scripts/deploy ของ ws6 ถ้าจำเป็น
3) ระบบ phase 1-WS5 ต้องยังทำงานได้ เพราะ ws6 ไม่แตะ core runtime
4) ถ้า backup script มีปัญหา ให้ปิด timer backup ก่อน
